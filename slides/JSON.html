Fundamento

Que es JSON

Antes de escribir JSON, necesitamos entender que es y para que sirve.

JSON (JavaScript Object Notation) es un formato ligero de intercambio de datos, facil de leer y escribir para humanos, y facil de procesar para maquinas.

Nacio en 2001 como un subconjunto de JavaScript, y en 2013 se convirtio en el estandar ECMA-404, independiente de cualquier lenguaje.

Hoy es el formato dominante para APIs REST Interfaz de programacion de aplicaciones , archivos de configuracion y almacenamiento de datos.

Sintaxis

Las 6 reglas de oro

Si sigues estas reglas al pie de la letra, tu JSON sera siempre valido.

1

Las claves SIEMPRE van entre comillas dobles

"nombre", "edad", "activo" — Nunca comillas simples, nunca sin comillas.

2

Pares clave-valor separados por dos puntos

"clave": valor — El dos puntos es el conector obligatorio.

3

Comas separan pares, NUNCA al final

Despues de cada par va una coma, excepto el ultimo. JSON no permite trailing commas.

4

Objetos con llaves, arreglos con corchetes

{ } para colecciones de pares clave-valor. [ ] para listas ordenadas de valores.

5

Sin comentarios, sin funciones, sin variables

JSON es puro dato. No existe // ni /* */, no hay funciones ni expresiones.

6

Solo 6 tipos de datos validos

String, Number, Boolean, null, Object, Array. Nada mas, nada menos.

Ejemplo Anotado

Un JSON completo, paso a paso

Lee cada linea con atencion: los colores indican el tipo de dato.

alumno.json
1{ 2 // Este es un OBJETO principal 3 "nombre": "Maria Garcia", 4 "edad": 21, 5 "esActivo": true, 6 "promedio": 9.5, 7 "carrera": null, 8 9 // Un ARREGLO de objetos anidados 10 "materias": [ 11 { 12 "nombre": "Programacion", 13 "calificacion": 10 14 }, 15 { 16 "nombre": "Matematicas", 17 "calificacion": 9 18 } 19 ] 20}
Nota: Los comentarios (//) en realidad NO son validos en JSON puro. Los usamos aqui solo con fines educativos. En JSON real, debes eliminarlos.

Tipos de Datos

Los 6 tipos que acepta JSON

JSON no admite fechas, funciones, undefined ni tipos personalizados. Solo estos seis.

"texto" String
42 Number
true Boolean
null Null
[ ... ] Array
{ ... } Object
Cuidado: undefined, NaN, Infinity, funciones ni comentarios son validos en JSON. Si tu lenguaje los genera, debes transformarlos antes de serializar.
Estructuras

Objeto vs Arreglo

Estos son los dos contenedores fundamentales de JSON. Entender la diferencia es clave.

{ } Objeto — Coleccion de pares clave-valor (sin orden garantizado)
objeto.json
1{ 2 "id": 1, 3 "nombre": "Calculo", 4 "creditos": 8 5}
Cada clave es unica. Si se repite, el ultimo valor sobreescribe a los anteriores.
[ ] Arreglo — Lista ordenada de valores (puede ser de cualquier tipo)
arreglo.json
1[ 2 "Calculo", 3 "Algebra", 4 "Programacion" 5]
Los indices empiezan en 0. No tiene claves, solo posicion. Puedes mezclar tipos dentro del mismo arreglo.
Anidamiento

Estructuras dentro de estructuras

La verdadera potencia de JSON esta en combinar objetos y arreglos a cualquier profundidad.

universidad.json
1{ 2 "universidad": "UNAM", 3 "carreras": [ // Arreglo de objetos 4 { 5 "nombre": "Ingenieria", 6 "semestres": 8, 7 "materias": [ // Arreglo dentro de objeto 8 { "clave": "IF001", "nombre": "Fisica" }, 9 { "clave": "IF002", "nombre": "Quimica" } 10 ] 11 } 12 ] 13}
Regla practica: Piensa en la estructura como un arbol. El objeto raiz es el tronco, los arreglos son las ramas, y los valores simples son las hojas. Evita mas de 4 niveles de profundidad para mantener legibilidad.
Buenas Practicas

Hazlo bien vs Hazlo mal

Compara los errores comunes con las formas correctas de escribir JSON.

Hazlo asi
Claves siempre con comillas dobles
"nombre": "Carlos"
Evita esto
Sin comillas o comillas simples
nombre: "Carlos" o 'nombre': "Carlos"
Hazlo asi
Sin coma despues del ultimo elemento
{ "a": 1, "b": 2 }
Evita esto
Trailing comma (coma al final)
{ "a": 1, "b": 2, ← ERROR }
Hazlo asi
Usa null para valores vacios
"telefono": null
Evita esto
undefined o vacio sin null
"telefono": undefined "telefono": "" (si no aplica)
Hazlo asi
Nombres de clave descriptivos y consistentes
"primerNombre": "Ana" "apellidoPaterno": "Lopez"
Evita esto
Claves ambiguas o inconsistentes
"nom": "Ana" "apellido_paterno": "Lopez" "ApellidoMaterno": ...
Hazlo asi
Fechas como strings en formato ISO 8601
"fechaNacimiento": "2003-05-15"
Evita esto
Fechas como objetos Date o formatos raros
"fecha": new Date() "fecha": "15/05/2003" (ambiguo)
